# 山本さん YouTube台本生成スキル

テーマを受け取り、Xリサーチ（任意）→ 台本構成 → RAG検索 → YouTube台本生成 を行う。
各ステップでユーザーの承認を得てから次に進む。
承認は必ず **AskUserQuestionツール** を使い、選択肢付きで確認すること。

> **X記事スキルからの連携時:** yamamoto-x-postスキルで「YouTube台本も作る」を選択した場合、
> すでにテーマ・Xリサーチ結果・記事構成・RAGデータが会話コンテキストにある。
> その場合はステップ①〜④をスキップし、ステップ⑤に直行する。
> 「X記事作成で収集したリサーチ・RAGデータを引き継ぎます。YouTube台本生成に進みます。」と表示してからステップ⑤を開始すること。

## 設定

- **insightsフォルダ**: `/Users/teach/environment/app/2507/260428_yamamoto-rag-x-main/insights/`
- **RAG仕訳インデックス**: `/Users/teach/environment/app/2507/260428_yamamoto-rag-x-main/insights/_routing.md`
- **台本プロンプトファイル**: `/Users/teach/environment/app/2507/260428_yamamoto-rag-x-main/insights/video_prompt/Louis Gleeson氏プロンプト.md`
- **Xリサーチ**: デフォルトON（ユーザーが「Xリサーチなし」「リサーチスキップ」等と言った場合はOFF）
- **取得量と反映量**: 取得は広く（5〜10ファイル）、反映は **2〜3件** に絞る（30分台本+Shorts3本でも同じ事例を軸に深く使う）

## データ枯渇防止の運用原則（厳守）

1. RAGデータは広く取得してよい（5〜10ファイル程度）
2. ただし1本の台本に反映する具体事例は **2〜3件** に絞ること（テーマ一貫性優先）
3. Shorts3本を含めても、同一事例を別角度で使い回す方針（新規事例を増やさない）
4. テーマとの整合性が高い事例のみ採用し、関連度の低いものは捨てる
5. **同じ `*_analysis.txt` の再利用**（頻度・切り口）は、ユーザーの方針または都度の指示に従う（リポジトリ側で使用履歴は持たない）

## ステップ② Xで最新情報をリサーチ

> Xリサーチ=OFFの場合はこのステップをスキップし、ステップ③も省略してステップ④へ進む。

**やること:**

3段階のフォールバック方式でXの最新情報を収集する。
各Phaseで十分な結果（具体的な投稿内容を含む3件以上）が得られたら、次のPhaseはスキップする。

### Phase 1: WebSearchで検索（無料）
1. WebSearchツールを使い、以下の検索クエリでXの最新投稿・トレンドを調査する:
   - `{テーマ} site:x.com`
   - `{テーマ} Twitter トレンド`
   - テーマに関連するキーワードの組み合わせ（2〜3パターン）
2. **判定:** 具体的な投稿内容を含む結果が3件以上 → Phase 2, 3をスキップ

### Phase 2: Nitterでフォールバック補完（無料）
3. Phase 1の結果が不十分な場合、Nitter経由で追加リサーチする:
   - WebFetchツールで `https://xcancel.com/search?q={テーマ}&f=tweets` を取得
   - 取得できない場合は `https://twiiit.com/search?q={テーマ}&f=tweets` を試す
   - それでも取得できない場合は `https://nitter.privacydev.net/search?q={テーマ}&f=tweets` を試す
4. **判定:** Phase 1 + 2 の合計で3件以上 → Phase 3をスキップ

### Phase 3: X API で最終補完（従量課金）
5. Phase 1 + 2 でも情報が不十分な場合、X APIで検索する:
   - 環境変数 `X_API_BEARER_TOKEN` が設定されているか確認
   - 未設定の場合: ユーザーに「X APIが未設定のため、現在の情報で進めます」と通知し、手持ちの情報で進める
   - 設定済みの場合: 以下のコマンドでツイート検索を実行
   ```bash
   curl -s -H "Authorization: Bearer ${X_API_BEARER_TOKEN}" \
     "https://api.x.com/2/tweets/search/recent?query={テーマ}&max_results=20&tweet.fields=created_at,public_metrics,text" \
     | python3 -c "
   import json, sys
   data = json.load(sys.stdin)
   for tweet in data.get('data', []):
       print(f\"[{tweet.get('created_at','')}] {tweet['text']}\")
       metrics = tweet.get('public_metrics', {})
       print(f\"  likes: {metrics.get('like_count',0)}, RT: {metrics.get('retweet_count',0)}\")
       print()
   "
   ```

6. 全Phaseの結果を統合し、以下のフォーマットでまとめる:

```
## Xリサーチ結果

情報ソース: {WebSearch / Nitter / X API}（使用したPhaseを記載）

### トレンド・話題
- {箇条書きで3〜5件}

### 注目の意見・切り口
- {箇条書きで3〜5件}

### 使えそうなフレーズ・表現
- {箇条書きで3〜5件}

### 高エンゲージメント投稿（X API使用時のみ）
- {いいね・RT数が多い投稿の要約}
```

## ステップ④ 山本さんRAGに投げる

**やること:**
1. insightsフォルダ内のファイルを検索し、テーマに関連する情報を抽出する
2. 以下のプロンプトの意図に沿って、RAGデータから情報を集める:

あなたはユーザーから投げられた{{ テーマ }}をもとに
そこに必要な情報を{insightsフォルダ}から情報を探し出し、適切な回答を作成してください

ある程度推論しても構わないので十分に検索してください
[/prompt]

**具体的な手順:**

### A. RAGデータを広めに取得（取得量は多め）
1. **最初に必ず** Readツールで `insights/_routing.md` を読む（grepより先に読むこと）
2. ユーザーテーマから、`_routing.md` のカテゴリを判定する
   - 主カテゴリは必ず **1つだけ** 選ぶ
   - 補助カテゴリは必要な場合のみ **0〜1つ** 選ぶ
   - 判定理由を1行でメモする
3. 主カテゴリの「優先して読むファイル」から **5〜8件** を先に読み込む
4. 補助カテゴリを選んだ場合は、補助カテゴリから最大2件だけ追加で読む
5. 優先ファイルだけで候補が足りない場合に限り、Grepツールで `insights/` 直下の `*.txt` を追加検索する
6. 各ファイルからテーマに関連する知見・具体例・数字・エピソードを抽出する

### B. 反映する具体事例を絞り込む（反映量は最小限）
7. 抽出した事例のうち、テーマとの整合性が最も高いものとして **採用候補2〜3件** に絞り込む
8. 採用事例は原則として主カテゴリから選ぶ。補助カテゴリの事例を採用する場合は1件までにする
9. 採用候補が2〜3件揃わない場合、キーワードを変えて追加のGrep検索を行う
10. Shortsで別事例を新たに追加しないこと（本編と同じ2〜3件を別角度で使う）

### C. 結果をまとめる
11. Xリサーチ結果（ある場合）と統合して、以下のフォーマットでまとめる:


## ステップ⑤ YouTube台本を生成

**やること:**
1. ステップ②〜④で集めた情報をすべて統合する
2. **Readツール**で台本プロンプトファイルを読み込む:
   - `/Users/teach/environment/app/2507/260428_yamamoto-rag-x-main/insights/video_prompt/Louis Gleeson氏プロンプト.md`
   - このファイルには以下が含まれている:
     - **BRAND DNA**: 山本智也の話し方・思考法・口調パターン
     - **YouTube台本テンプレート**: 30分構成・12セクション（フック〜CTA）
     - **離脱防止設計**: 5分ごとのつなぎフレーズ
     - **サムネイル＆タイトル生成ルール**
     - **Shorts台本テンプレート**: 3パターン（数字インパクト型・How-to型・共感→転換型）
     - **出力フォーマット**: 統一フォーマット
     - **品質チェックリスト**: 21項目

3. 読み込んだプロンプトに**忠実に**従い、以下の成果物を一括生成する:

### 生成する成果物

**A. タイトル案（3パターン）**
- プロンプトのタイトル生成ルールに従う
- 28〜40文字、数字を1つ以上、SEOキーワードを前半に
- パターンA〜Eのいずれかで生成

**B. サムネテキスト案（5パターン）**
- 10〜18文字、数字を大きく見せる
- 感情ワードを入れる（「衝撃」「激変」「完全版」等）
- 色指示を含める

**C. 動画構成表（30分）**
- 12セクションの時間・内容一覧表

**D. 台本本文（9,000〜10,000文字）**
- プロンプトの12セクション構成に忠実に従う:
  - SECTION 1: フック（0:00〜0:20）
  - SECTION 2: テーマ発表＋自己紹介＋全体像（0:20〜1:30）
  - SECTION 3: 問題提起＋背景解説（1:30〜5:00）
  - SECTION 4: 中間まとめ①（5:00〜5:30）
  - SECTION 5: 本編パート1 — 概念・考え方（5:30〜11:00）
  - SECTION 6: 体験談・エピソード（11:00〜13:00）
  - SECTION 7: 中間まとめ②＋後半予告（13:00〜13:30）
  - SECTION 8: 本編パート2 — 実践・How-to（13:30〜22:00）
  - SECTION 9: Q&A・よくある質問（22:00〜25:00）
  - SECTION 10: 応用・発展（25:00〜27:00）
  - SECTION 11: 全体まとめ（27:00〜28:30）
  - SECTION 12: CTA＋次回予告（28:30〜30:00）
- 各セクションに [テロップ案] [画面切替指示] を付ける
- すべて**話し言葉**で書く（書き言葉厳禁）
- BRAND DNAの口調パターンを忠実に再現する

**E. Shorts台本（3本）**
- Shorts①: 数字インパクト型（60秒）
- Shorts②: How-to型（60秒）
- Shorts③: 共感→転換型（60秒）

### 台本生成時の重要ルール

1. **RAGデータ活用**: ステップ④で抽出した山本さんの具体的な数字・エピソード・実体験を必ず織り込む
2. **話し言葉**: すべて話し言葉。「〜ですね」「〜じゃないですか」「めちゃめちゃ」「ぶっちゃけ」等のBRAND DNA口調を使う
3. **数字の裏付け**: 具体的な数字を最低8箇所に含める
4. **二部構成**: 前半＝考え方・フレームワーク、後半＝実践・How-to
5. **離脱防止**: 5分ごとに中間まとめ・つなぎフレーズを入れる
6. **禁止表現**: 「簡単に稼げます」系の煽り、根拠のない断言、説明なしの横文字専門用語、ネガティブな締め、競合名指し否定

### 品質チェック（生成後・表示前に必ず実施）

プロンプトファイルの品質チェックリスト21項目を確認する:
- 合計文字数が9,000〜10,000文字か
- フック冒頭3秒に数字 or 結論が入っているか
- 山本トーンが自然に入っているか
- 具体的な数字が最低8箇所に含まれているか

## 注意事項

- 各ステップで必ず **AskUserQuestionツール** でユーザーの承認を得てから次に進むこと（テキストで「よろしいですか？」と聞くのではなく、選択肢UIを使う）
- Xリサーチ=OFFの場合、ステップ②③をスキップしてステップ④に進む
- X記事スキルからの連携時は、ステップ①〜④をスキップしてステップ⑤に直行する
- 台本プロンプトファイルは必ずReadツールで読み込んでから台本を生成すること
- 台本は9,000〜10,000文字（話し言葉）、12セクション構成
- すべて話し言葉で書く。書き言葉を混ぜない
- 山本さんのRAGデータから具体的な数字やエピソードを積極的に活用する（ただし **1本あたり具体事例は2〜3件** まで）
- 取得は広く（5〜10ファイル）、反映は絞る（2〜3件）を徹底する
- Shorts3本は **本編と同じ2〜3件を別角度で使い回す**（新規事例を追加しない）
- `insights/` の元ファイルは移動・削除しない
- X記事スキル連携時（ステップ⑤直行）も、採用事例2〜3件への絞り込みは **必ず実施** する
- Shorts台本3本（数字型・How-to型・共感型）を必ず同時生成する
- 品質チェックリストを通過してからユーザーに表示すること
