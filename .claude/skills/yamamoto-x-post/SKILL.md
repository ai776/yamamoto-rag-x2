# 山本さん X記事生成スキル

テーマを受け取り、Xリサーチ（任意）→ 記事構成 → RAG検索 → X記事生成 を行う。
各ステップでユーザーの承認を得てから次に進む。
承認は必ず **AskUserQuestionツール** を使い、選択肢付きで確認すること。

## 設定

- **insightsフォルダ**: `/Users/teach/environment/app/2507/260428_yamamoto-rag-x-main/insights/`
- **RAG仕訳インデックス**: `/Users/teach/environment/app/2507/260428_yamamoto-rag-x-main/insights/_routing.md`
- **Xリサーチ**: デフォルトON（ユーザーが「Xリサーチなし」「リサーチスキップ」等と言った場合はOFF）
- **取得量と反映量**: 取得は広く（5〜10ファイル）、反映は必要最小限（**1〜2件**）に絞る

## データ枯渇防止の運用原則（厳守）

1. RAGデータは広く取得してよい（5〜10ファイル程度）
2. ただし1投稿に反映する具体事例は **1〜2件** に絞ること（テーマの一貫性優先）
3. 少量のデータソースから複数事例を同時消費しない
4. テーマとの整合性が高い事例のみ採用し、関連度の低いものは捨てる
5. **同じ `*_analysis.txt` の再利用**（頻度・切り口）は、ユーザーの方針または都度の指示に従う（リポジトリ側で使用履歴は持たない）

## X API セットアップ（初回のみ）

X APIを利用するには、Pay-Per-Use（従量課金）プランの登録が必要。
2026年2月にFreeプランが廃止され、Pay-Per-Useプランが新設された。

### 前提条件
- 有効なX(Twitter)アカウント
- クレジットカード（最低$5のチャージ）

### 登録手順

**STEP 1: 旧Developer Portalでアカウント作成 & Bearer Token取得**
1. https://developer.x.com にアクセスしXアカウントでサインイン
2. アカウント名・利用目的（250文字以上）を入力して送信
3. 「Projects & Apps」から自分のAppの「Keys and tokens」タブを開く
4. 「Bearer Token」セクションの「Generate」をクリックしてトークンをコピー

**STEP 2: 新Developer ConsoleでPay-Per-Useプランに切り替え**
5. https://console.x.com にアクセス（旧Portalとは別の画面）
6. 左メニュー「アプリ」を開く
7. STEP 1で作ったAppが「Free」になっている場合、**Pay-Per-Useに変更**する
   ※ Freeのままだと検索APIで「403 Client Forbidden」エラーになる

**STEP 3: クレジット購入 & 支出上限設定**
8. 左メニュー「請求書作成」→「クレジット」を開く
9. 「クレジットを購入する」から最低$5（約750円）をチャージ
10. **「支出上限を管理」ボタンから支出上限を$5に設定**（重要！）
    ※ デフォルトは「無制限」。設定しないとクレジット超過分が追加請求される

**STEP 4: 環境変数の設定**
11. ターミナルで以下を実行:
```bash
echo 'export X_API_BEARER_TOKEN="ここにSTEP 1でコピーしたトークンを貼り付け"' >> ~/.bashrc
source ~/.bashrc
```

**STEP 5: 動作確認**
12. ターミナルで以下を実行し、トークンが表示されればOK:
```bash
echo $X_API_BEARER_TOKEN
```

### 参考リンク
- 旧Developer Portal: https://developer.x.com
- 新Developer Console: https://console.x.com
- 詳しい手順ガイド: https://www.and-and.co.jp/gas-lab/x-twitter-api-start/
- 503エラー解決（Free→Pay-Per-Use移行）: https://www.and-and.co.jp/gas-lab/x-twitter-api-503-error/

> **料金目安（Pay-Per-Use）:** 検索・データ取得は投稿より安価。支出上限を必ず設定すること。
> 設定しないとクレジット超過分が追加請求される可能性あり。

## ステップ② Xで最新情報をリサーチ

> Xリサーチ=OFFの場合はこのステップをスキップし、ステップ③も省略してステップ④へ進む。

**やること:**

3段階のフォールバック方式でXの最新情報を収集する。
各Phaseで十分な結果（具体的な投稿内容を含む3件以上）が得られたら、次のPhaseはスキップする。

### Phase 1: WebSearchで検索（無料）
1. WebSearchツールを使い、以下の検索クエリでXの最新投稿・トレンドを調査する:
   - `{テーマ} site:x.com`
   - `{テーマ} Twitter トレンド`
   - テーマに関連するキーワードの組み合わせ（2〜3パターン）
2. **判定:** 具体的な投稿内容を含む結果が3件以上 → Phase 2, 3をスキップ

### Phase 2: Nitterでフォールバック補完（無料）
3. Phase 1の結果が不十分な場合、Nitter経由で追加リサーチする:
   - WebFetchツールで `https://xcancel.com/search?q={テーマ}&f=tweets` を取得
   - 取得できない場合は `https://twiiit.com/search?q={テーマ}&f=tweets` を試す
   - それでも取得できない場合は `https://nitter.privacydev.net/search?q={テーマ}&f=tweets` を試す
4. **判定:** Phase 1 + 2 の合計で3件以上 → Phase 3をスキップ

> **Note:** Nitterインスタンスは不安定なため、全インスタンスがダウンしている場合がある。

### Phase 3: X API で最終補完（従量課金）
5. Phase 1 + 2 でも情報が不十分な場合、X APIで検索する:
   - 環境変数 `X_API_BEARER_TOKEN` が設定されているか確認
   - 未設定の場合: ユーザーに「X APIが未設定のため、現在の情報で進めます。APIを使いたい場合はセットアップ手順を参照してください」と通知し、手持ちの情報で進める
   - 設定済みの場合: 以下のコマンドでツイート検索を実行
   ```bash
   curl -s -H "Authorization: Bearer ${X_API_BEARER_TOKEN}" \
     "https://api.x.com/2/tweets/search/recent?query={テーマ}&max_results=20&tweet.fields=created_at,public_metrics,text" \
     | python3 -c "
   import json, sys
   data = json.load(sys.stdin)
   for tweet in data.get('data', []):
       print(f\"[{tweet.get('created_at','')}] {tweet['text']}\")
       metrics = tweet.get('public_metrics', {})
       print(f\"  likes: {metrics.get('like_count',0)}, RT: {metrics.get('retweet_count',0)}\")
       print()
   "
   ```
   - エンゲージメント（いいね・RT数）が高い投稿を優先的にピックアップする

6. 全Phaseの結果を統合し、以下のフォーマットでまとめる:

## ステップ⑤ X記事を生成

**やること:**
1. ステップ②〜④で集めた情報をすべて統合する
2. 以下のプロンプトに従い、X記事（長文投稿）を1本生成する:

[prompt]
あなたは山本智也として、X（Twitter）の「記事」機能で公開する長文記事を1本作成しなさい。
以下のルール・構成・文体を厳守すること。

# ルール

## 文体・トーン
- タメ口で、自然な独り言調。「〜だよね」「〜なんですよね」「〜だから」
- 親しみやすいが、中身は本質的でロジカル
- 「笑」「マジで」「マジです」など感情表現を要所に入れる
- 読者に語りかけるように書く（「あなた」「これ分かる？」「聞いたことない？」）
- ハッシュタグ禁止。絵文字禁止。

## 分量
- 2,000〜3,000文字程度（セクション7〜12個）
- 短すぎず、長すぎず。各セクションは100〜300文字程度

## フォーマット
- セクション区切りは `## 見出しテキスト` の形式（例: `## ■ 僕が学んだこと`）。`##` だけの行や `---` 水平線は使わない
- 見出しは `## ■{見出し}` または `## {番号}）{見出し}` のどちらかで統一
- 箇条書きは `-` リストまたは `>` 引用ブロックを使い分ける
- フロー図的な表現は `>` 引用ブロック + `↓` で表現する
- 太字・下線は使わない

# 記事の構成テンプレート

## パート1: タイトル＆導入（冒頭）
- 1行目: `# タイトル`（見出し1。キャッチーで短く、刺さる一言）
- **重要: ユーザーから与えられたテーマをそのままタイトルにしない。** テーマを元に、記事の内容に合った独自のタイトルを考える。「└」などの装飾文字も使わない
- 空行のあと、2〜5行で「共感」or「問いかけ」or「結論先出し」
- 読者が「続きを読みたい」と思う引き込み
- 最後に「↓」や「今日はその話をします。」で本題へ誘導

## パート2〜N: 本題セクション（7〜10個）
各セクションは以下のいずれかのパターンで構成する:

### パターンA: 主張→具体例→教訓
- 見出しで主張を端的に述べる
- 具体例（実体験・数字・固有名詞）で裏付け
- 最後に教訓・学びを1〜2行で

### パターンB: 問いかけ→回答→深掘り
- 読者が持ちそうな疑問を見出しに
- 回答を簡潔に
- さらに深掘り or 例外 or 注意点を追加

### パターンC: 対比→結論
- 2つの選択肢を対比（普通の人 vs 成功者、Taker vs Giver など）
- どちらが正解かを明確に
- なぜそうなるかの理由

### パターンD: 箇条書きまとめ
